## Project Overview

EvalAI is an AI-powered answer sheet evaluation platform built for school teachers. It automatically grades typed student PDF answer sheets against a question paper and answer key using the Groq LLM API. The project was built iteratively over multiple sessions with AI assistance for code generation and human guidance for architecture decisions and debugging.

## Manual Modifications Made

These are changes made manually after identifying bugs through debugging sessions:

### 1. Answer Key Parsing Bug Fix
**Problem:** The answer key for Q20 was being stored as `D\n\n-- 1 of 1 --` instead of just `D` because the PDF page marker was included.

**Manual Fix in `questionParser.js`:**
```javascript
// Added after extracting answer key value
answerKey = answerKey.trim().charAt(0).toUpperCase();
```

### 2. Student Answer Extraction Fix
**Problem:** Student answers formatted as `Answer: B ✓` or `Answer: B ✗` were not being parsed correctly. The parser was extracting `B ✓` instead of just `B`.

**Manual Fix in `answerParser.js`:**
```javascript
// Changed regex to extract only the letter
const match = line.match(/Answer:\s*([A-D])/i);
if (match) {
    answer = match[1].toUpperCase();
} else if (line.includes('Answer:')) {
    answer = 'No answer provided';
}
```

### 3. Groq JSON Key Variation Fix
**Problem:** Groq API was returning different JSON key names across calls — sometimes `evaluation_results`, sometimes `student_evaluations`, sometimes `evaluations`, causing the parser to return empty arrays and skip saving scores.

**Manual Fix in `groqEvaluator.js`:**
```javascript
// Added fallback chain for all observed key variations
const results = parsed.evaluation_results 
  || parsed.students_evaluations 
  || parsed.student_evaluations
  || parsed.evaluations
  || parsed.evaluation
  || parsed.results
  || [];
```

### 4. Total Marks Calculation Fix
**Problem:** The teacher-entered `total_marks` value was being used for percentage calculation, but the actual sum of question marks from the question paper was different. This caused percentages above 100%.

**Manual Fix in `exam.js` after parsing questions:**
```javascript
const totalMarksFromQuestions = questions.reduce((sum, q) => sum + q.marks, 0);
db.prepare('UPDATE exams SET total_marks = ? WHERE id = ?')
  .run(totalMarksFromQuestions, examId);
```

### 5. MCQ Evaluation Not Saving Bug
**Problem:** MCQ grading was comparing answers correctly but the UPDATE statement was never executing — every question showed `BEGIN → COMMIT` with no updates in between, resulting in all students getting 0.

**Manual Fix:** Identified that the marks comparison result was not being passed to the database update function. Fixed by ensuring the `marks_awarded` variable was correctly scoped and passed to the UPDATE query.

### 6. API_BASE Not Defined Error
**Problem:** After adding the Exam History feature, `API_BASE` was accidentally removed from `Dashboard.jsx`, causing the dashboard to crash with `ReferenceError: API_BASE is not defined`.

**Manual Fix:** Created a centralized `frontend/src/config.js`:
```javascript
export const API_BASE = 'http://localhost:5000';
```
And imported it in all page components.

## Challenges Faced

### Challenge 1 — PDF Parsing Inconsistency
Different student PDF templates had different answer formats. Some used `Answer: B ✓`, others used `Answer: B ✗`, others just `Answer: B`, and blank sheets used `Answer: ____`. Each format required separate handling in the regex parser.

**Resolution:** Built a flexible regex `/Answer:\s*([A-D])/i` that extracts just the letter regardless of what follows it, with a fallback for blank answers.

### Challenge 2 — Groq API Non-Deterministic JSON Keys
The Groq LLM model would return different JSON key names for the same prompt across different calls. This caused silent failures where evaluation results were parsed as empty arrays and never saved to the database.

**Resolution:** Implemented a fallback chain that checks multiple possible key names before giving up, and added a warning log when no results are parsed.

### Challenge 3 — Scores Exceeding Maximum Marks
The system was showing scores like 51.5/40 because the teacher entered 40 as total marks but the actual question paper had 55 marks worth of questions (10 MCQ + 7 theory questions).

**Resolution:** Changed the system to auto-calculate total marks by summing all question marks after parsing, overwriting the teacher-entered value with the correct computed total.

### Challenge 4 — Rate Limiting on Groq Free Tier
With 3 students and 17 questions, some evaluation batches were hitting Groq's rate limits mid-evaluation, causing silent failures where questions were skipped entirely.

**Resolution:** Added retry logic with a 10-second delay on rate limit errors, and added the `GROQ RAW RESPONSE` debug log to identify which questions were being skipped.

### Challenge 5 — SQLite ALTER TABLE Migration
When adding new columns (`is_overridden`, `class_account_id`) to existing tables, SQLite does not support `IF NOT EXISTS` on `ALTER TABLE`, causing server crashes on startup if columns already existed.

**Resolution:** Wrapped all `ALTER TABLE` statements in try/catch blocks so they silently skip if the column already exists.

### Challenge 6 — Route Conflicts After Feature Addition
After adding the Exam History page, existing dashboard routes stopped working because a new route was conflicting with existing ones in Express. The `GET /api/exams` route was being matched before `GET /api/exam/:id/results`.

**Resolution:** Reordered routes in `index.js` to ensure more specific routes are registered before generic ones.

## Lessons Learned

**1. Test each service in isolation before wiring to routes.**
Every time a new service (PDF parser, answer parser, Groq evaluator) was built, it should have been tested standalone first. Several bugs were only discovered after the full pipeline was connected, making them harder to trace.

**2. LLMs are non-deterministic — never assume consistent output format.**
The Groq model changing JSON key names taught an important lesson: always write defensive parsers for LLM output. Never assume the model will return exactly what the prompt asked for. Always have fallbacks.

**3. Debug logs are essential during development, remove them before production.**
Adding `console.log('RAW PDF TEXT:', text)` and `console.log('PARSED ANSWERS:', answers)` at each pipeline stage was the fastest way to find bugs. These should always be removed before pushing to production.

**4. Database migrations need careful handling in SQLite.**
SQLite is simple but has limitations — no `ALTER TABLE IF NOT EXISTS`, no easy column type changes. Plan the schema fully upfront to avoid painful migrations later.

**5. The teacher-entered value cannot always be trusted.**
Relying on the teacher to enter the correct total marks introduced a class of bugs. Computing values from actual data (summing question marks) is always more reliable than trusting user input.

**6. Centralize configuration from the start.**
The `API_BASE` bug that broke the dashboard after adding a new feature could have been avoided entirely if a central config file had been created from day one. Never scatter magic strings across components.

**7. Git commit frequently with clear messages.**
During debugging sessions, it was difficult to track which version of the code was working. Committing after each working feature with a clear message would have made rollbacks much easier.
